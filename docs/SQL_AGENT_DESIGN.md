## Architecture Overview

### High-Level Flow
```
START
  ↓
list_tables (discover available tables)
  ↓
get_schema (fetch table structure)
  ↓
generate_query (LLM writes SQL)
  ↓
[Conditional] → check_query (validate SQL) OR END
  ↓
run_query (execute SQL)
  ↓
analyze_results (interpret data)
  ↓
END
```

### Key Design Decisions

**1. Why LangGraph over LangChain?**
- LangGraph provides explicit state management and deterministic workflow control
- Better error handling with conditional edges
- Easier to debug and test individual nodes
- Supports human-in-the-loop patterns for future extensions

**2. Query Validation Strategy**
- Always validate generated SQL before execution
- Use a specialized LLM call with error-checking prompt
- Catch common SQL errors (NULL handling, UNION vs UNION ALL, type mismatches)
- Prevents database errors and improves reliability

**3. Analysis Capabilities**
- Row counts: "How many MSAs are in the dataset?"
- Aggregations: "What's the average wage in Austin?"
- Trends: "Show wage growth from 2000-2023"
- Comparisons: "Compare Austin vs San Francisco wages"
- Filtering: "Which MSAs have over 100k avg wage?"

## State Definition

### Core State Schema
```python
from typing import TypedDict, List, Optional
from typing_extensions import Annotated

class SQLAgentState(TypedDict):
    """
    State for SQL agent workflow.

    This state is passed between nodes and accumulates information
    throughout the workflow execution.
    """
    # Input
    user_query: str  # Original natural language question

    # Database discovery
    available_tables: List[str]  # Tables discovered in database
    table_schema: str  # Schema information for relevant tables

    # Query generation
    generated_sql: Optional[str]  # SQL generated by LLM
    sql_valid: bool  # Whether SQL passed validation
    validation_error: Optional[str]  # Error message if validation failed

    # Query execution
    query_results: Optional[List[dict]]  # Results from database
    execution_error: Optional[str]  # Error if query failed

    # Analysis
    analysis: str  # Natural language interpretation of results

    # Metadata
    num_retries: int  # Track retry attempts for query generation
    messages: List[dict]  # Message history for LLM context
```

### Why This State Structure?

1. **Separation of Concerns**: Each phase has dedicated fields
2. **Error Tracking**: Explicit error fields for debugging
3. **Retry Logic**: Counter prevents infinite loops
4. **Transparency**: All intermediate steps preserved for logging

## Node Implementations

### Node 1: List Tables

**Purpose**: Discover available tables in the database

```python
from langchain_core.messages import AIMessage
from langgraph.graph import MessagesState

def list_tables(state: SQLAgentState) -> dict:
    """
    Programmatically invoke the sql_db_list_tables tool.

    This node doesn't use the LLM - it directly calls the tool
    to ensure we always check available tables first.
    """
    # Create a synthetic tool call
    tool_call = {
        "name": "sql_db_list_tables",
        "args": {},
        "id": "list_tables_call",
        "type": "tool_call",
    }

    # Invoke the tool
    list_tables_tool = next(
        tool for tool in tools
        if tool.name == "sql_db_list_tables"
    )
    tool_message = list_tables_tool.invoke(tool_call)

    # Parse results
    tables = tool_message.content.split(", ")

    # Create response message
    response = AIMessage(
        content=f"Available tables: {tool_message.content}"
    )

    return {
        "available_tables": tables,
        "messages": [AIMessage(content="", tool_calls=[tool_call]),
                     tool_message, response]
    }
```

**Key Points**:
- No LLM call (saves latency and cost)
- Programmatic tool invocation
- Updates state with table list
- Returns messages for LLM context

### Node 2: Get Schema

**Purpose**: Fetch table structure for the relevant table(s)

```python
from langgraph.prebuilt import ToolNode

# Create specialized ToolNode for schema fetching
get_schema_tool = next(
    tool for tool in tools
    if tool.name == "sql_db_schema"

get_schema_node = ToolNode([get_schema_tool], name="get_schema")

def call_get_schema(state: SQLAgentState) -> dict:
    """
    Wrapper that calls get_schema_node with the detected table.

    For this specific use case, we know the table is
    "msa_wages_employment_data", but this pattern allows
    for dynamic table selection in the future.
    """
    # Create tool call for the target table
    tool_call = {
        "name": "sql_db_schema",
        "args": {"table_names": "msa_wages_employment_data"},
        "id": "get_schema_call",
        "type": "tool_call",
    }

    # Invoke schema tool
    tool_message = get_schema_tool.invoke(tool_call)

    return {
        "table_schema": tool_message.content,
        "messages": [
            AIMessage(content="", tool_calls=[tool_call]),
            tool_message
        ]
    }
```

**Design Note**: This could be enhanced to:
- Parse user query to detect which tables are relevant
- Fetch multiple table schemas if needed
- Cache schema to avoid repeated lookups

### Node 3: Generate Query

**Purpose**: Use LLM to translate natural language to SQL

```python
from langchain_anthropic import ChatAnthropic

# Initialize LLM
llm = ChatAnthropic(model="claude-sonnet-4-5-20250929", temperature=0)

# System prompt for SQL generation
SYSTEM_PROMPT = """
You are an expert PostgreSQL query writer for QCEW employment and wage data.

TASK: Generate a syntactically correct PostgreSQL query to answer the user's question.

DATABASE SCHEMA:
Table: msa_wages_employment_data
Columns:
- area_fips: MSA identifier code (TEXT)
- year: Year (INTEGER, 2000+)
- qtr: Quarter (TEXT, 'A' = Annual average)
- size_code: Establishment size code (TEXT)
- size_title: Size category description (TEXT)
- annual_avg_estabs_count: Number of establishments (INTEGER)
- annual_avg_emplvl: Employment level (INTEGER)
- total_annual_wages: Total wages (BIGINT)
- avg_annual_pay: Average annual pay (INTEGER)
- annual_avg_wkly_wage: Average weekly wage (INTEGER)
- area_title: MSA name, e.g., "Austin-Round Rock, TX" (TEXT)
- state: State abbreviation (TEXT)

IMPORTANT RULES:
1. ALWAYS use qtr = 'A' for annual data (most common use case)
2. For MSA names, use ILIKE for case-insensitive matching:
   WHERE area_title ILIKE '%Austin%'
3. ALWAYS limit results to 100 rows maximum unless user specifies
4. Use proper aggregation functions (SUM, AVG, COUNT, MIN, MAX)
5. For trend analysis, ORDER BY year ASC
6. NEVER use DELETE, UPDATE, INSERT, DROP, or other DML/DDL statements
7. Double-check your query for common errors before returning

EXAMPLES:
Q: "What's the average wage in Austin in 2023?"
A: SELECT area_title, avg_annual_pay, year
   FROM msa_wages_employment_data
   WHERE area_title ILIKE '%Austin%' AND year = 2023 AND qtr = 'A'
   LIMIT 100;

Q: "Show wage trends for Austin from 2010-2020"
A: SELECT year, avg_annual_pay
   FROM msa_wages_employment_data
   WHERE area_title ILIKE '%Austin%' AND qtr = 'A'
     AND year BETWEEN 2010 AND 2020
   ORDER BY year ASC;

Q: "Compare average wages between Austin and San Francisco in 2022"
A: SELECT area_title, AVG(avg_annual_pay) as avg_wage
   FROM msa_wages_employment_data
   WHERE (area_title ILIKE '%Austin%' OR area_title ILIKE '%San Francisco%')
     AND year = 2022 AND qtr = 'A'
   GROUP BY area_title;
"""

def generate_query(state: SQLAgentState) -> dict:
    """
    Generate SQL query using LLM with schema context.
    """
    # Build context from state
    user_query = state["user_query"]
    table_schema = state.get("table_schema", "")

    # Prepare messages
    messages = [
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user", "content": f"Schema:\n{table_schema}\n\nQuestion: {user_query}"}
    ]

    # Add previous messages if this is a retry
    if state.get("validation_error"):
        messages.append({
            "role": "assistant",
            "content": f"Previous attempt failed: {state['validation_error']}"
        })

    # Bind the sql_db_query tool for structured output
    run_query_tool = next(
        tool for tool in tools
        if tool.name == "sql_db_query"
    )
    llm_with_tool = llm.bind_tools([run_query_tool], tool_choice="any")

    # Generate query
    response = llm_with_tool.invoke(messages)

    # Extract SQL from tool call
    if response.tool_calls:
        tool_call = response.tool_calls[0]
        sql_query = tool_call["args"]["query"]

        return {
            "generated_sql": sql_query,
            "messages": state["messages"] + [response]
        }
    else:
        return {
            "generated_sql": None,
            "execution_error": "LLM did not generate a tool call",
            "messages": state["messages"] + [response]
        }
```

**Critical Design Points**:
- Temperature = 0 for deterministic SQL generation
- Schema-aware prompt with examples
- Safety rules against DML/DDL
- Retry context included for failed attempts

### Node 4: Check Query (Validator)

**Purpose**: Validate generated SQL before execution

```python
CHECK_QUERY_PROMPT = """
You are a SQL query validator. Review the query for common errors:

1. NOT IN with NULLs: NOT IN with NULL values always returns empty set
2. UNION vs UNION ALL: Use UNION ALL unless you need deduplication
3. BETWEEN for exclusive ranges: BETWEEN is inclusive on both ends
4. Data type mismatches: Ensure comparisons use compatible types
5. Incorrect quoting: PostgreSQL identifiers need double quotes, strings single quotes
6. Function syntax: Verify function arguments and types
7. Aggregation errors: Ensure all non-aggregated columns are in GROUP BY

If the query has errors, rewrite it correctly.
If the query is correct, return it unchanged.
"""

def check_query(state: SQLAgentState) -> dict:
    """
    Validate the generated SQL query using a second LLM call.

    This is a best practice from LangGraph docs to catch errors
    before database execution.
    """
    generated_sql = state.get("generated_sql")

    if not generated_sql:
        return {
            "sql_valid": False,
            "validation_error": "No SQL query to validate"
        }

    # Prepare validation messages
    system_message = {"role": "system", "content": CHECK_QUERY_PROMPT}
    user_message = {"role": "user", "content": generated_sql}

    # Bind run_query_tool to force tool call format
    run_query_tool = next(
        tool for tool in tools
        if tool.name == "sql_db_query"
    )
    llm_with_tool = llm.bind_tools([run_query_tool], tool_choice="any")

    # Validate
    response = llm_with_tool.invoke([system_message, user_message])

    # Check if query was rewritten
    if response.tool_calls:
        validated_sql = response.tool_calls[0]["args"]["query"]

        # Compare to original
        is_changed = validated_sql.strip() != generated_sql.strip()

        return {
            "generated_sql": validated_sql,
            "sql_valid": True,
            "validation_error": "Query rewritten by validator" if is_changed else None,
            "messages": state["messages"] + [response]
        }
    else:
        return {
            "sql_valid": False,
            "validation_error": "Validator did not produce valid output"
        }
```

**Why This Matters**:
- Catches 80% of common SQL errors before execution
- Reduces database error rate
- Provides better error messages to user
- Second LLM call adds ~200ms but prevents failures

### Node 6: Analyze Results

**Purpose**: Convert SQL results to natural language

```python
ANALYSIS_PROMPT = """
You are a data analyst interpreting query results.

TASK: Provide a clear, concise natural language answer to the user's question
based on the query results.

GUIDELINES:
1. Start with a direct answer to the question
2. Include relevant numbers and statistics
3. Mention any notable patterns or insights
4. If results are empty, explain why
5. Keep it conversational and easy to understand
6. Use proper formatting (bullet points, numbers) for readability

EXAMPLE:
Question: "What's the average wage in Austin in 2023?"
Results: [{"area_title": "Austin-Round Rock, TX", "avg_annual_pay": 68450, "year": 2023}]
Analysis: "In 2023, the average annual pay in the Austin-Round Rock, TX metro area was $68,450."
"""

def analyze_results(state: SQLAgentState) -> dict:
    """
    Generate natural language analysis of query results.
    """
    user_query = state["user_query"]
    query_results = state.get("query_results", [])
    generated_sql = state.get("generated_sql", "")

    # Handle empty results
    if not query_results:
        return {
            "analysis": "No data found matching your query. This could mean:\n"
                       "- The MSA name was misspelled\n"
                       "- No data exists for the specified time period\n"
                       "- The filter criteria were too restrictive"
        }

    # Format results for LLM
    results_str = "\n".join([str(row) for row in query_results[:50]])  # Limit for token efficiency

    messages = [
        {"role": "system", "content": ANALYSIS_PROMPT},
        {"role": "user", "content": f"Question: {user_query}\n\n"
                                   f"SQL Query: {generated_sql}\n\n"
                                   f"Results:\n{results_str}\n\n"
                                   f"Provide analysis:"}
    ]

    # Generate analysis
    response = llm.invoke(messages)

    return {
        "analysis": response.content,
        "messages": state["messages"] + [response]
    }
```

### Conditional Logic: Should Continue?

**Purpose**: Decide whether to validate query or end workflow

```python
from typing import Literal
from langgraph.graph import END

def should_continue(state: SQLAgentState) -> Literal["check_query", END]:
    """
    Route decision after generate_query.

    - If LLM generated a query with tool call → validate it
    - If no tool call → end workflow with error
    """
    messages = state["messages"]
    last_message = messages[-1]

    # Check if last message has tool calls
    if not hasattr(last_message, "tool_calls") or not last_message.tool_calls:
        return END

    return "check_query"
```

## Error Handling & Safety

### 1. SQL Injection Prevention

**Strategy**: Use parameterized queries via SQLAlchemy

```python
# BAD - vulnerable to injection
query = f"SELECT * FROM users WHERE name = '{user_input}'"

# GOOD - parameterized (handled by LangChain tools)
# The QuerySQLDataBaseTool uses SQLAlchemy text() with proper escaping
```

**Additional Safeguards**:
- Database user has READ-ONLY permissions (set in PostgreSQL)
- Tool descriptions explicitly forbid DML/DDL statements
- System prompt reinforces SELECT-only queries

### 2. Database Connection Management

```python
from sqlalchemy import create_engine, pool

# Connection pooling configuration
engine = create_engine(
    DB_URI,
    poolclass=pool.QueuePool,
    pool_size=5,  # Max concurrent connections
    max_overflow=10,  # Burst capacity
    pool_timeout=30,  # Wait time for connection
    pool_recycle=3600,  # Recycle connections after 1 hour
    pool_pre_ping=True,  # Verify connections before use
)
```

### 3. Query Timeout Protection

```python
# Set statement timeout in PostgreSQL connection string
DB_URI = f"{base_uri}?options=-c statement_timeout=30000"  # 30 seconds

# Or set per-session in the database
conn.execute(text("SET statement_timeout = 30000"))
```

### 4. Result Size Limits

```python
def run_query_safe(state: SQLAgentState) -> dict:
    """
    Execute query with size limit.
    """
    # Ensure LIMIT clause exists
    sql = state["generated_sql"]
    if "LIMIT" not in sql.upper():
        sql = sql.rstrip(";") + " LIMIT 100;"

    # Execute with timeout
    try:
        results = run_query_tool.invoke({"query": sql})
        return {"query_results": results, "execution_error": None}
    except Exception as e:
        return {"query_results": None, "execution_error": str(e)}
```

### 5. Retry Logic

```python
MAX_RETRIES = 3

def should_retry(state: SQLAgentState) -> Literal["generate_query", "analyze_results"]:
    """
    Retry query generation if validation fails.
    """
    num_retries = state.get("num_retries", 0)

    if state.get("execution_error") and num_retries < MAX_RETRIES:
        return "generate_query"  # Try again with error context
    elif state.get("query_results"):
        return "analyze_results"  # Success, proceed
    else:
        return END  # Give up after max retries
```

### 6. Error Messages for Users

```python
def format_error_for_user(error: str) -> str:
    """
    Translate technical errors to user-friendly messages.
    """
    if "syntax error" in error.lower():
        return "I generated an invalid SQL query. Let me try again."
    elif "column" in error.lower() and "does not exist" in error.lower():
        return "I referenced a column that doesn't exist. Checking schema..."
    elif "timeout" in error.lower():
        return "Query took too long. Try narrowing your date range or filtering by MSA."
    else:
        return f"Database error: {error}"
```

### Phase 4: Build Graph (1 hour)

**Step 4.1**: Create agent.py

```python
# sql_agent/agent.py
from langgraph.graph import StateGraph, START, END
from langgraph.prebuilt import ToolNode
from .state import SQLAgentState
from .nodes import (
    list_tables,
    call_get_schema,
    generate_query,
    check_query,
    analyze_results,
    should_continue,
)
from .tools import tools

# Get specialized tool nodes
get_schema_tool = next(t for t in tools if t.name == "sql_db_schema")
get_schema_node = ToolNode([get_schema_tool], name="get_schema")

run_query_tool = next(t for t in tools if t.name == "sql_db_query")
run_query_node = ToolNode([run_query_tool], name="run_query")

# Build graph
builder = StateGraph(SQLAgentState)

# Add nodes
builder.add_node("list_tables", list_tables)
builder.add_node("call_get_schema", call_get_schema)
builder.add_node("get_schema", get_schema_node)
builder.add_node("generate_query", generate_query)
builder.add_node("check_query", check_query)
builder.add_node("run_query", run_query_node)
builder.add_node("analyze_results", analyze_results)

# Add edges
builder.add_edge(START, "list_tables")
builder.add_edge("list_tables", "call_get_schema")
builder.add_edge("call_get_schema", "get_schema")
builder.add_edge("get_schema", "generate_query")

# Conditional edge: validate if query generated, else end
builder.add_conditional_edges(
    "generate_query",
    should_continue,
    {
        "check_query": "check_query",
        END: END
    }
)

builder.add_edge("check_query", "run_query")
builder.add_edge("run_query", "analyze_results")
builder.add_edge("analyze_results", END)

# Compile
agent = builder.compile()

# Export for use
def run_agent(user_query: str) -> dict:
    """
    Run the SQL agent with a natural language query.

    Args:
        user_query: Natural language question about the data

    Returns:
        Dictionary with analysis, SQL query, and results
    """
    initial_state = {
        "user_query": user_query,
        "available_tables": [],
        "table_schema": "",
        "generated_sql": None,
        "sql_valid": False,
        "validation_error": None,
        "query_results": None,
        "execution_error": None,
        "analysis": "",
        "num_retries": 0,
        "messages": [],
    }

    result = agent.invoke(initial_state)

    return {
        "query": user_query,
        "sql": result.get("generated_sql"),
        "results": result.get("query_results"),
        "analysis": result.get("analysis"),
        "error": result.get("execution_error"),
    }
```

## Common Pitfalls and Solutions

### Pitfall 1: Case-Sensitive MSA Names
**Problem**: "austin" doesn't match "Austin-Round Rock, TX"
**Solution**: Always use ILIKE in WHERE clauses

### Pitfall 2: Missing LIMIT Clause
**Problem**: Query returns too many rows, timeout
**Solution**: Inject LIMIT 100 if not present

### Pitfall 3: Wrong Quarter Filter
**Problem**: User asks for "2023" but qtr != 'A'
**Solution**: Always add qtr = 'A' unless user specifies quarterly data

### Pitfall 4: State Type Errors
**Problem**: TypedDict doesn't enforce types at runtime
**Solution**: Add validation in nodes:
```python
def validate_state(state: SQLAgentState):
    assert isinstance(state.get("user_query"), str)
    assert isinstance(state.get("num_retries"), int)
```

## Glossary

**Node**: A function in the graph that processes state and returns updates
**Edge**: Connection between nodes that defines execution flow
**State**: Dictionary-like object passed between nodes
**Tool**: Pre-built function (like SQL query execution) that nodes can invoke
**Conditional Edge**: Decision point that routes to different nodes based on state
**ToolNode**: Special node type that executes LangChain tools
